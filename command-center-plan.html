03The two genuinely new pieces

A · Pi as an engine

Pi ships a full SDK (session.prompt() + streaming) and a headless RPC mode. Add a pi_engine next to codex.rs / gemini.rs — ideally an SDK sidecar so every profile gets the real llm-wiki skill + extensions. One engine, many profiles — the persona, model, skills and memory vault differ per division; the runtime is shared.

B · The delegation protocol

The heart of it — structured messages, not free chat. Every arrow is one typed row in cc_messages. The Slack/Discord feel is just a view over that log.

The part you care most about

04Memory — brains that grow

One LLM Wiki vault per profile under a shared parent, plus a thin org-index for the General. After every mission the profile runs wiki_retro into its vault — that's literally "the division gets stronger."

command-center/
├─ org/wiki/                 # General's org-wide index · mission history · links out
└─ divisions/
   ├─ design/wiki/
   ├─ marketing/wiki/
   ├─ research/wiki/
   └─ dev/wiki/
↻ The growth loop
Reusable skills get promoted into the Control Panel Skill Library — so any profile, any division, can equip what one division learned.
◉ The Obsidian view
Reuses Learn's forceLayout.ts physics + wiki backlinks. Per-vault graph, switchable, with a whole-org union mode.

05Data model

A profile is a persona + brain model + skills + memory vault + rank. Missions are tracked as a per-profile list (the Kanban's job, absorbed) — no swarm fan-out, so no cc_agents table.

-- 0027_command_center.sql (append-only · supersedes Hermes board)
cc_profiles    id, name, rank(commander|general|captain), division, accent,
               brain_model, skill_ids, wiki_root, charter, autonomy_level
cc_channels    id, kind(command|division|cross|dm), division?, name
cc_messages    id, channel_id, from_profile_id, to_profile_id?,
               kind(chat|directive|report|handoff), body, mission_ref, ts
cc_missions    id, title, brief, status, autonomy_level,
               assigned_profile_id, origin_profile_id, ts
Ship gated · graduate later

06The autonomy ladder

Same approval pattern ROSIE/Hermes already use — so it's just a column, not new code. Default everything to level 1, per division.

LEVEL 0
Manual
Nothing fires without a human click.
LEVEL 1 default
Approve each step
Agent plans; you approve every dispatch.
LEVEL 2
Auto within budget
Runs to a token / task ceiling, then auto-stops.
LEVEL 3
Full auto + digest
Runs free; you read the briefing after.

08Master brief

COMMAND CENTER — MASTER BRIEF. Build a new top-level Orion app (gold accent) that replaces the Hermes Kanban: an AI org you command. One engine (pi, via SDK sidecar), many profiles. A profile = persona + brain model + skills + its own LLM-Wiki vault + rank. Ranks: Commander (you) → General (pure coordinator) → Captain (one per division: Design/Marketing/Research/Dev…). No operatives, no swarms — each Captain works serially in its own session. Surface is channel-first (Slack-like: command / per-division / cross-division / DM) over a typed cc_messages log; delegation = directive→report→briefing messages, approval-gated (autonomy ladder, default L1). Each profile's vault grows via wiki_retro; reusable skills promote into the Control Panel library; an Obsidian-style graph view reuses Learn's forceLayout.ts. Migration 0027, append-only (cc_profiles/cc_channels/cc_messages/cc_missions); keep hermes_tasks read-only. Reuse Agent Forge, provider runtime, activity_log, ROSIE's gated-planning pattern. Per-phase protocol: research → audit → ranked plan (approval) → green TDD slices → user smoke test. First slice: migration + profiles model + read-only org/channel shell, before any profile runs.
